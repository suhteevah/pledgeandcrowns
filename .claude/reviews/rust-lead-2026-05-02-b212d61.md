## P1 (fix this week)

### [`compile-api/src/cargo_grader.rs:177-187`] `cargo check` inherits parent env — strip it

`Command::new("cargo")` is spawned with no `.env_clear()` and no targeted `.env_remove(...)`. The parent process's env propagates wholesale: `RUSTFLAGS`, `CARGO_TARGET_DIR`, `CARGO_HOME`, `RUSTC_WRAPPER`, `CARGO_BUILD_JOBS`, network proxies, locale, etc. None of these are *player-controlled* today, but two concrete failure modes are already in scope:

1. Matt's repo sets `CARGO_TARGET_DIR=G:/cargo-target/pledgeandcrown` via `.cargo/config.toml`. That env var leaks into the player sandbox build, meaning every player submission's `target/` lands in the shared workspace target dir instead of the disposable sandbox. The `SandboxGuard` then can't clean it up — it only removes `<temp>/pledge-cargo-grader/<uuid>/`.
2. `RUSTC_WRAPPER=sccache` (or any future wrapper Matt installs globally) becomes a third-party process running on every player submission with no review.

Fix: explicit allowlist. Spawn with `.env_clear()` then re-add only `PATH`, `CARGO_HOME`, `RUSTUP_HOME`, and a sandbox-local `CARGO_TARGET_DIR=<sandbox>/target`. While there, set `.current_dir(&sandbox)` so `--manifest-path` redundancy is also defended at the CWD layer (currently CWD is whatever the server happened to be launched from).

### [`compile-api/src/cargo_grader.rs:91-103`] No source-size cap on `compile_check`

`run_cargo_check` bounds *output* (16 KiB stderr) but not *input*. A player POSTs a 10 MB source string; the lib.rs write succeeds, then `cargo check` parses 10 MB of garbage into rustc and we eat the parse-time cost. The HTTP route (`compile_real`) only checks `is_empty()`. Add a cap (suggest 64 KiB — design/01 missions are <2 KiB) at the route level OR top of `compile_check`, returning a setup-time `Err`. Same cap should apply to `/compile` for symmetry.

### [`game/src/plugins/stub_grader.rs:46-330` ↔ `compile-api/src/grader.rs:33-316`] Pattern-table duplication: keep, but add a parity test

Recommendation: **keep the duplication** for now. The trade-off:

- Extracting a `pledge_grader_core` crate adds a workspace edge `game -> grader_core` that pulls AGPL into the MIT game binary unless the new crate is dual-licensed; doing the dual-license dance for an internal crate has a cost. It also adds ~5-15s to the cold pre-commit cycle on the G: target dir.
- Duplication is fine *if* drift is mechanically detected. We've already had one drift incident (`190fdda`). Today's `stub_covers_every_mission_in_the_registry` checks coverage, not behavior parity.

Concrete fix (low cost, high value): add a contract test in `game/tests/contract.rs` that boots `make_router()` from `pledge_compile_api`, sends each canonical solution to `/compile`, and asserts the server's verdict (ok flag + stderr-shape-without-`[stub]`-prefix) matches `stub_verdict(...)` byte-for-byte after stripping the prefix. The HTTP harness already exists at `compile-api/tests/http.rs`; reuse the pattern. That turns "we have two pattern tables" from a known-fragile latent bug into a single source of truth enforced at CI time.

If a second drift incident lands despite the parity test, escalate to extraction. Not before.

### [`compile-api/src/lib.rs:46-133`] `compile_real` happy-path runs the pattern grader on top of cargo's verdict — semantic confusion

`compile_real` calls `cargo_grader::compile_check` (real) and on success ALSO calls `grader::grade(...)` (pattern). That means a player who submits source that compiles but doesn't match the per-encounter pattern gets `ok: true` from cargo and a *failing* `pattern.stdout` flavor message stitched into the response — except the response's `ok` field is taken from cargo, not the pattern grader. So the response says "ok=true, the Smith eyes you — missing required: `let mut`". That's a contradictory verdict.

Pick one of:
- (a) `ok = verdict.ok && pattern.ok`. The route is now a curriculum-faithful grader.
- (b) Drop the pattern call entirely — `/compile-real` means "did this compile?" and that's it. The flavor text is the v0 grader's job; once `/compile-real` is the default, retire the pattern grader and write per-mission canonical-output tests instead.

(b) is cleaner and matches the design's eventual end-state. (a) is a one-line shim if you want `/compile-real` usable as a drop-in for `/compile` today. Either way, the current shape is wrong.

## P2 (taste, not urgent)

### [`compile-api/src/lib.rs`] Route handler size — not yet over the threshold

`lib.rs` is ~140 LOC, two routes, one is a 50-line handler. Below the 150-LOC blob threshold from the role file. Don't split into `routes/` yet. Promote to a split when either: (1) a third non-trivial route lands, or (2) `compile_real` itself gets multi-stage logic (cache lookup → cargo → wasmtime → response). At that point the right shape is `routes/{compile,compile_real,health}.rs` with `lib.rs` keeping only `make_router` + the request/response types.

### [`compile-api/src/grader.rs:42-316`] Pattern table: not yet time for a DSL

600 LOC of repetitive arms reads worse than it grades. Each arm averages ~10 LOC, follows the same shape (`if !contains(needle) { fail(format!(...)) } else { ... pass(flavor) }`), and the flavor text is the most variable part — exactly what makes a table-driven approach fragile (a `&[(&str, &[&str], &str)]` table forces all flavor strings into a single shape and trades easily-readable per-arm logic for harder-to-debug data).

A small DSL is justified when: (a) we cross ~30 missions with the same pattern shape, OR (b) we want non-engineers (Matt? voice-editor agent?) to add missions without touching Rust. We're at 21 missions and Matt is the author. Hold.

When the DSL does land, the right shape is probably:

```rust
struct PatternRule { needles: &'static [&'static str], err: &'static str }
struct Mission { id: &'static str, rules: &'static [PatternRule], pass: &'static str }
static MISSIONS: &[Mission] = &[ ... ];
```

with `grade()` reduced to a lookup + linear rule scan. That's a 2026-Q3 refactor, not a 2026-05 one.

### [`compile-api/src/wasm_runner.rs:125`] Defence-in-depth: deny more wasm features

Only `wasm_threads(false)` is set. `wasmtime::Config` exposes a dozen feature toggles; for the curriculum-grade path, also explicitly set `.wasm_simd(false)`, `.wasm_reference_types(false)`, `.wasm_bulk_memory(true)` (Rust toolchains emit it; can't disable), `.wasm_tail_call(false)`, `.wasm_function_references(false)`, `.wasm_gc(false)`, `.wasm_component_model(false)`. None of these are P0 — the runtime is already bounded on fuel/epoch/memory — but explicit denies turn "future wasmtime release silently enables a feature" from a possible regression into a compile error.

### [`compile-api/src/wasm_runner.rs:164-170`] Watchdog uses `thread::sleep` — non-cancellable

If the wasm finishes in 2 ms but `timeout_ms = 30000`, the watchdog thread sits parked for 30 s before joining. Functionally fine (the `done` flag suppresses the epoch tick) but wastes a thread per submission and is visible to `ps`/Task Manager under load. Swap to a `Condvar` or `crossbeam_channel::recv_timeout` so the fast path joins immediately. Performance-only; ignore until profiling shows it.

### [`compile-api/Cargo.toml:36-37`] Wasmtime versions

`wasmtime = "44"` + `wasmtime-wasi = { version = "44", features = ["p1"] }` is the current stable pair (Cargo.lock pins 44.0.1, released ~April 2026). Versions and lockstep are correct. Note for future: when bumping, both crates MUST move together — they share internal types.

### [`compile-api/Cargo.toml:42`] `wat` dev-dep is justified

`wat = "1"` only appears under `[dev-dependencies]` and is used by `compile-api/tests/wasm_runner.rs:21` to synthesize wasm bytes inline. The alternative — committing precompiled `.wasm` blobs — would mean (a) binary blobs in the repo, (b) regenerating them when test cases change, (c) a separate toolchain invocation. `wat` is a single small crate maintained by the bytecodealliance (same org as wasmtime). Keep.

### [`compile-api/Cargo.toml:30`] `uuid` features

`uuid = { version = "1", features = ["v4"] }`. Minimal, correct, no `serde`/`js`/`fast-rng` features pulled in. Good.

### [`compile-api/src/cargo_grader.rs:198-199`] `let _ = tx;` placeholder

`let (tx, rx) = mpsc::channel::<()>(); let _ = tx;` keeps `tx` alive only by accident — the comment says so but the code reads as dead. Either drop the channel entirely (the loop only uses `rx.recv_timeout` as a sleep, which `thread::sleep(Duration::from_millis(50))` would do without the import) or delete it. Pure noise reduction.

### [`compile-api/src/cargo_grader.rs:192-193`] `expect("stdout was piped")` on `child.stdout.take()`

These two `.expect`s are justified — `Stdio::piped()` was set on the same builder five lines up, so `take()` returning `None` is genuinely unreachable. Per role-file rules, `expect` outside of tests usually warrants a callout, but the precondition is local and obvious. No change needed; flagging only because the reviewer's checklist asks.

## Praise

- **`wasm_runner.rs` security contract is documented in the module header before any code runs.** Reads like a security review checklist; future reviewers can validate "is this still true?" against a paragraph rather than a 200-LOC scan. Replicate this pattern when `compile-api` grows a third sensitive module.
- **`SandboxGuard` Drop-based cleanup in `cargo_grader.rs:65-83`** is the right shape — RAII handles the success, error, and panic-unwind cases without an explicit cleanup at every return site. Better than the typical `defer!` macro pattern Matt's seen in other Rust services.
- **`classify_trap` in `wasm_runner.rs:210-273`** correctly distinguishes WASI `I32Exit(0)` from `OutOfFuel` from `Interrupt` from generic traps and turns each into a player-readable verdict. The "treat fuel/timeout as `Ok(verdict { ok: false })` not `Err`" call (documented in the module header) is exactly right — they're grading outcomes, not server bugs, and treating them as `Err` would have leaked them into 500-status HTTP responses.
- **Test isolation discipline.** The slow `cargo_grader` integration tests are `#[ignore]`-gated so the pre-commit hook stays fast (~40s), and the wasm_runner tests use inline `wat` so they don't depend on a `wasm32-wasip1` toolchain being installed on Matt's box. Both are the right local maxima for the team's CI shape.
- **The `[stub]` prefix contract** in `stub_grader.rs:31-43` plus the load-bearing assertion in `tests/stub_grader.rs:168-172` is a good example of "make the invariant a UI-visible thing AND a test-asserted thing". Replicate this whenever a verdict path branches.
